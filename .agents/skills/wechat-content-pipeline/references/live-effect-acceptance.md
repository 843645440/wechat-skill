# 真实效果验收：用一篇草稿替代继续代码审查

## 触发条件

当用户明确要求“不再根据代码审”“直接测试一篇”“按实际效果审”时，切换到本流程：

- 冻结代码，不再继续静态审计、补测试、提交或推送。
- 不修改全局配置，不碰未授权账号。
- 用一篇真实文章跑完整流水线，以最终草稿、封面、正文图和预览为证据。

这不是常规生产任务，也不是 dry-run。目标是验证真实端到端效果，因此只有用户明确授权时才创建草稿；始终止于草稿箱，不公开发布。

## 隔离工作区

为避免覆盖账号日常任务，可使用独立临时基目录：

```bash
BASE=$(mktemp -d /tmp/wechat-live-effect-XXXXXX)
python3 <PIPELINE_ROOT>/scripts/pipeline_job.py init \
  --project-root <PROJECT_ROOT> \
  --work-dir "$BASE" \
  --account <AUTHORIZED_TEST_ACCOUNT> \
  --topic "<TEST_TOPIC>"
```

注意：`--work-dir` 接受的是基目录，`init` 实际返回的任务路径是：

```text
<BASE>/<account>/current/job.json
```

后续命令必须使用 `init` 输出的真实 `job.json` 路径，不能把 `$BASE` 直接当任务目录。不要使用 `--force-new`，除非用户明确授权丢弃同一隔离目录中的旧任务。

## 一篇真实稿的执行范围

1. 使用给定主题；未给主题时选一个边界清楚、可由权威来源核验的代表性主题。
2. 正文、来源、一次 strong humanize、1—3 张正文图、随机主题、确定性封面、严格 HTML 校验全部照常执行。
3. 完整流水线已由账号档案和触发请求解析正文配图数量、后端、语言与输出目录时，不要因下游 Baoyu Skill 的独立使用首次设置而擅自创建全局 `EXTEND.md`，也不要重复向用户询问。上游流水线配置覆盖独立交互设置；仍须遵守 Baoyu 的提示词先落盘、图片真实生成和视觉检查契约。
4. 正文图阶段必须准确写入：

```bash
python3 <PIPELINE_ROOT>/scripts/pipeline_job.py stage \
  --job <JOB> --name illustrations --status completed \
  --detail image_count=<实际张数> \
  --detail skill=baoyu-article-illustrator
```

`prepare` 读取的是 `details.image_count`；`count` 或把多个键塞进一个 `count=...` 字符串都无效。门禁失败时先修本轮任务元数据，确认尚未创建草稿，再重跑 `prepare`。
4. `prepare` 成功后才运行正式 `finish`。一次只启动一个 `finish` 进程。

## 以像素和落盘状态验收

不要因生成工具返回 success、CLI exit 0 或草稿 API 返回 media_id 就直接宣称效果合格。按顺序检查：

### 正文图（插入前）

逐张用视觉工具查看真实像素并核对：

- 所有中文、数字、引号和流程标签逐字正确；
- 无乱码、重复字符、多余英文、色值文字或水印；
- 构图、留白、信息层级适合公众号正文；
- 文件真实落盘，横向尺寸合理；
- 只有通过检查的图才插入 `article.md`。

### 封面（finish 后）

检查实际 `cover/cover.png`，不能只信尺寸和探针：

- 标题逐字正确且无裁切；
- 长中文标题是否出现字距过紧、字形重叠或字号过大；
- 小图预览下仍可读；
- 对比度、构图和账号调性是否合格。

`night-signal` 等大字模板遇到约 18 字或两行中文标题时，要特别检查视觉拥挤；“未越界”不等于“视觉上合格”。

### 完整正文预览

通过临时本地 HTTP 服务打开 `article_preview.html`，按实际页面检查：

- 段落字号、行距、留白和主题配色；
- 小标题、编号、分隔线是否抢正文；
- 正文图片宽度、边距和手机阅读清晰度；
- 是否缺图、溢出、乱码或样式塌陷；
- “本文看点”等摘要组件是否完整反映正文结构。

验收后关闭临时 HTTP 服务。

## 草稿后的安全边界

一旦正式 `finish` 已创建草稿：

- 先核验 `job.state=drafted`、`draft.status=completed` 和非空 `draft_media_id`。
- 发现封面或排版仅有视觉瑕疵时，保留当前草稿并如实报告。
- 不要为了“顺手修好”自动修改产物并再次运行正式 `finish`；这可能产生第二份草稿。只有用户明确要求重做时才开始新一轮，并说明旧草稿仍存在。
- 效果报告聚焦真实结果：目标账号、标题、字数、主题、正文图数量、草稿状态，以及正文/封面各自的优缺点。不要重新转回代码审查。

## 用户只要求封面复测时

若完整真实草稿已经存在，用户随后明确要求“不要全量测，只测封面图”，立即收窄范围：

1. 不重跑 `prepare` 或正式 `finish`，不重新渲染正文，不创建或更新微信草稿。
2. 保留已有草稿和 `cover/cover.png` 作为基线；候选封面输出到独立路径，避免让本地任务状态看起来已经替换了线上草稿。
3. 只比较注册模板和标题排印的实际像素。若多个模板都出现同类中文粘连，停止盲目换模板，转入 `wechat-html-cover` 的 [references/cjk-title-legibility.md](../../wechat-html-cover/references/cjk-title-legibility.md) 做受控单图收敛。
4. 标题文字、正文和事实性副文案保持不变；只允许针对单张本地产物调整字号、字重、字距和标题块位置。
5. 最终只报告封面尺寸、文字准确性、裁切/粘连/缩略图可读性，以及“草稿是否更新”。用户未授权替换草稿时，必须明确说明当前微信草稿仍使用原封面。
