# 原生信息模块计划格式

计划文件使用 UTF-8 JSON。它是内部事实保护和插入定位记录，不进入公众号正文。

## 顶层结构

```json
{
  "version": 1,
  "theme": "olive-journal",
  "modules": []
}
```

`modules` 允许 0—3 项。没有自然成立的结构时保留空数组。

## 通用字段

每个模块必须包含：

```json
{
  "id": "inline-01",
  "kind": "insight",
  "title": "效率提高之后，压力转移给了谁",
  "placement": {
    "after_heading": "被自动化改变的首先是工作分配",
    "after_text": "企业节省的是流程时间，但员工承担了新的复核责任。"
  },
  "evidence": [
    "企业节省的是流程时间，但员工承担了新的复核责任。"
  ]
}
```

- `after_heading` 必须等于文章某个 `##` 标题，不含 `##`。
- `after_text` 必须是该章节中的原文连续片段。
- `evidence` 包含 1—4 段原文连续片段，只用于内部核对。
- 标题最多 24 字；同篇模块类型和插入位置不得重复。

## insight

增加 `items`，包含 2—4 项：

```json
"items": [
  {"label": "一线员工", "text": "重复操作减少，但复核责任增加"},
  {"label": "技术主管", "text": "需要重新划分工具权限和责任"}
]
```

`label` 最多 10 字，`text` 最多 42 字。

## comparison

增加 `left` 和 `right`：

```json
"left": {"heading": "演示阶段", "items": ["任务边界清楚", "失败成本较低"]},
"right": {"heading": "生产环境", "items": ["输入持续变化", "责任必须追溯"]}
```

两侧标题最多 12 字，各含 2—4 项，每项最多 28 字。两侧必须使用可比较的同一维度。

## process

增加 `steps`，包含 3—5 项：

```json
"steps": [
  {"label": "选场景", "text": "找到高频且边界清楚的任务"},
  {"label": "接数据", "text": "处理权限、格式和更新机制"},
  {"label": "做复核", "text": "明确人工检查和异常处理"}
]
```

`label` 最多 10 字，`text` 最多 28 字。步骤编号由主题组件生成。

## metrics

只在正文存在已核实数据时使用，增加 `metrics`，包含 2—4 项：

```json
"metrics": [
  {"value": "37%", "label": "任务时间下降", "note": "对应公开测试中的指定场景"},
  {"value": "3个月", "label": "观察周期", "note": "不是长期趋势预测"}
]
```

`value` 最多 12 字且必须原样出现在文章中；`label` 最多 12 字，`note` 最多 28 字。无法确认精确值时不要使用本类型。
