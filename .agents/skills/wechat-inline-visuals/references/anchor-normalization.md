# 信息模块锚点规范化

## 典型误判

正文可能包含行内 Markdown：

```markdown
**基层获得年轻的专业力量，毕业生获得进入真实工作现场的机会**，这是计划最有价值的连接。
```

计划中的可见纯文本锚点为：

```text
基层获得年轻的专业力量，毕业生获得进入真实工作现场的机会，这是计划最有价值的连接。
```

二者文字和标点相同，只是正文含 `**`。若校验器直接搜索原始 Markdown，会错误降级为空计划。

## 持久规则

- 计划中的 `after_text` 和 `evidence` 使用读者可见的纯文本。
- 校验器和渲染器必须采用同一规范化：仅移除成对包裹文本的 `<u>...</u>`、`**...**`、`__...__`、反引号、`==...==`、`++...++`，再折叠空白。孤立或字面使用的运算符和标点必须保留，例如 `C++`、`a == b`、邮箱中的 `++`。
- 规范化只能移除展示标记；不得改标点、同义改写、模糊匹配或语义匹配。
- 锚点应短、连续、唯一，优先完整短句；不要复制一整段，也不要概括原意。
- 计划失败仍只降级一次，不引入生成—校验循环。

## 回归验证

1. 测试先证明：纯文本锚点对包含行内 Markdown 的正文会被旧逻辑拒绝。
2. 修复后，`validate_plan` 应返回 `module_count: 1`、`degraded: false`。
3. 用同一文章和计划运行 `render_article.py`，确认模块标题及条目实际出现在 HTML。
4. 最后运行 `validate_gzh_html.py` 和完整测试套件。

这个测试应与“真正不存在的锚点仍被拒绝”同时保留，防止修复滑向宽松匹配。
